# EP-002 — Knowledge Base Centralizzata (soli-projects hub)

## Obiettivo

Completare il ciclo di vita della wiki (lint, syntheses, runbooks), integrare soli-projects come submodule nei repository verticali, e consolidare il task manager come punto unico di gestione cross-progetto.

## Contesto

soli-projects è configurato come KB centralizzata per 17 repository (16 soli92 + 1 collaborazione esterna), con pattern llm-wiki++ v2.11 [^src: raw/soli-projects-agents.md §Project overview]. L'ingest ha prodotto 26 pagine wiki. L'app web espone /wiki (navigator) e /tasks (task manager). Mancano: ciclo lint/heal, pagine synthesis/runbook, adozione submodule nei repo consumer.

### Policy collaborazioni esterne

I repository non soli92-owned (es. [[agentic-value-investor-application]], owner marcociullo86) sono indicizzati nella KB e tracciati nel kanban ma con flusso **read-only**: soli-projects ingerisce wiki e traccia task/epiche/storie per governance, ma **non scrive** verso il repo esterno (niente directives, niente submodule push, niente webhook ingest via soli-prof). Submodule (US-005) e directives si applicano solo ai repo soli92.

## Storie

- [[US-004]] — Completamento ciclo wiki: lint, syntheses, runbooks
- [[US-005]] — Integrazione submodule nei repository verticali
- [[US-006]] — Task manager unificato: Supabase + kanban factory

## Rischi

- Submodule overhead per sviluppatori (update manuale, conflitti merge).
- Wiki stale se il ciclo ingest non viene eseguito periodicamente.
- Doppia gestione task (pm_todos Supabase vs kanban markdown) senza sincronizzazione.
- Collaborazioni esterne: sync manuale (no webhook automatico); rischio di raw stale se non aggiornato.
