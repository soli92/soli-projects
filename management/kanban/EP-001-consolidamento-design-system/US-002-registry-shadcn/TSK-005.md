# TSK-005 — Validare pipeline registry:sync e registry:build

## Descrizione

Eseguire npm run registry:sync e npm run registry:build nel repo solids dopo un aggiornamento di componente, verificare che l'output sia coerente e che i consumer possano consumare i componenti aggiornati.

## Osservazioni da TSK-004 (audit 2026-06-25)

L'audit TSK-004 ha evidenziato le seguenti criticita' rilevanti per questa task:

### Componenti mai usati (52 su 57)
Il 91% del registry non viene usato da alcun consumer. Non e' necessario validare la pipeline di sync per tutti i componenti: prioritizzare quelli effettivamente adottati (avatar, button, input, tabs, textarea in soli-dm-fe).

### Componente custom non nel registry (anomalia)
`soli-dm-fe/components/ui/full-screen-loader.tsx` e' presente nella directory `components/ui/` ma non fa parte del registry SoliDS. Rischio: se la pipeline registry:sync sovrascrive `components/ui/`, questo file potrebbe essere perso o causare conflitti.

### Disallineamento style in soli-dm-fe
Il `components.json` di soli-dm-fe usa `style: "default"`, mentre il registry SoliDS e' definito con `style: "new-york"`. La pipeline registry:sync deve gestire questa discrepanza per evitare che i componenti aggiornati dal registry vengano applicati con lo stile errato.

### Pattern di consumo: nessun import diretto dal pacchetto npm
Nessun consumer importa componenti React direttamente con `import { X } from "@soli92/solids"`. soli-dm-fe usa installazione locale (shadcn CLI copy). La pipeline registry:sync deve quindi aggiornare i file copiati in `components/ui/`, non gli import del pacchetto npm.

### Priorita' validazione
Validare in ordine: button, input, tabs, textarea, avatar — unici componenti con utilizzo reale in soli-dm-fe.
