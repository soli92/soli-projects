# TSK-005 — Validare pipeline registry:sync e registry:build

## Descrizione

Eseguire npm run registry:sync e npm run registry:build nel repo solids dopo un aggiornamento di componente, verificare che l'output sia coerente e che i consumer possano consumare i componenti aggiornati.

## Osservazioni da TSK-004 (audit 2026-06-25)

L'audit TSK-004 ha evidenziato le seguenti criticita' rilevanti per questa task:

### Componenti mai usati (52 su 57)
Il 91% del registry non viene usato da alcun consumer. Non e' necessario validare la pipeline di sync per tutti i componenti: prioritizzare quelli effettivamente adottati (avatar, button, input, tabs, textarea in soli-dm-fe).

### Componente custom non nel registry (anomalia)
`soli-dm-fe/components/ui/full-screen-loader.tsx` e' presente nella directory `components/ui/` ma non fa parte del registry SoliDS. Rischio: se la pipeline registry:sync sovrascrive `components/ui/`, questo file potrebbe essere perso o causare conflitti.

### Disallineamento style in soli-dm-fe
Il `components.json` di soli-dm-fe usa `style: "default"`, mentre il registry SoliDS e' definito con `style: "new-york"`. La pipeline registry:sync deve gestire questa discrepanza per evitare che i componenti aggiornati dal registry vengano applicati con lo stile errato.

### Pattern di consumo: nessun import diretto dal pacchetto npm
Nessun consumer importa componenti React direttamente con `import { X } from "@soli92/solids"`. soli-dm-fe usa installazione locale (shadcn CLI copy). La pipeline registry:sync deve quindi aggiornare i file copiati in `components/ui/`, non gli import del pacchetto npm.

### Priorita' validazione
Validare in ordine: button, input, tabs, textarea, avatar — unici componenti con utilizzo reale in soli-dm-fe.

## Risultati validazione

Validazione eseguita il 2026-06-25.

### Esecuzione `npm run registry:sync`
- Completato con successo.
- Output: `registry/solids/` con 59 directory di componenti + `registry/registry.json` generato.
- Il passo sync fa mirror da `src/` ed e' autonomo (non richiede `shadcn` installato).

### Esecuzione `npm run registry:build` (passo `shadcn build`)
- **Problema riscontrato**: `shadcn` non e' installato nelle `node_modules` locali del repo solids, sebbene sia listato come devDependency (`"shadcn": "^4.1.0"`). Il comando fallisce con `sh: shadcn: command not found`.
- **Workaround**: eseguendo tramite `npx shadcn build --output ./registry/r` (versione 4.11.0 dalla cache npx globale) il build completa con successo.
- **Output prodotto** (`registry/r/`, 5 file):
  - `registry.json` — indice dei 4 bundle (10 KB)
  - `solids-button.json` — bundle button standalone (2.4 KB)
  - `solids-icons.json` — bundle icone (29 KB)
  - `solids-ui.json` — bundle completo componenti UI (201 KB, il piu' grande)
  - `solids-utils.json` — utility CSS/token (567 B)

### Disallineamento `style` confermato
- `soli-dm-fe/components.json`: `"style": "default"`, `"baseColor": "slate"`, `"rsc": true`
- `solids/components.json` (registry source): `"style": "new-york"`, `"baseColor": "slate"`, `"rsc": false`
- **Rischio**: shadcn CLI applica varianti di classi diverse per `default` vs `new-york` (es. `rounded-md` vs `rounded-lg`, spacing leggermente diverso). I componenti copiati da soli-dm-fe tramite `shadcn add @solids/...` vengono generati con lo stile `new-york` del registry, non `default`. Non causano errori runtime ma producono varianti visive inconsistenti rispetto al resto del progetto soli-dm-fe se altri componenti sono stati generati con `style: default`.
- **Azione raccomandata**: allineare `soli-dm-fe/components.json` a `style: "new-york"` oppure mantenere `default` e documentare che i 5 componenti SoliDS adottati potrebbero avere leggere differenze stilistiche.

### Rischio `full-screen-loader.tsx` custom
- `soli-dm-fe/components/ui/full-screen-loader.tsx` e' un componente custom non presente nel registry SoliDS.
- Se si esegue `shadcn add` o un'operazione di sync che sovrascrive la directory `components/ui/`, questo file puo' essere eliminato silenziosamente.
- **Azione raccomandata**: spostare il componente fuori da `components/ui/` (es. `components/custom/`) oppure aggiungerlo a `.shadcn-ignore` (se supportato), documentando esplicitamente che non e' gestito dal registry.

### Nota sulla `shadcn` in devDependencies
- Aprire issue o eseguire `npm install` nel repo solids per assicurarsi che `shadcn` sia presente in `node_modules`. Attualmente la pipeline `registry:build` dipende da `shadcn` nel PATH di sistema, il che e' fragile in CI.
- Fix consigliato: nel package.json, sostituire `shadcn build` con `npx --yes shadcn build` o assicurarsi che `npm ci` installi le devDependencies in CI.
